# kuma-lab プロジェクトルール

## デプロイ前の動作確認（厳守）

**ビルド成功はデプロイ許可ではない。** `vite build` や `dotnet build` の成功は、ランタイムエラーがないことを保証しない。

### デプロイ前に必ず行うこと

1. **ローカルでdev serverを起動**し、ブラウザで対象ページを開いて**画面が表示されること**を目視確認する
2. dev serverのHMRキャッシュが壊れている場合は `.svelte-kit` を削除して再起動し、**新しいブラウザタブ**で確認する
3. ブラウザで確認できない場合（真っ暗になる等）は、**絶対にデプロイしない**。原因を特定してから進める
4. 変更を1つずつ適用し、**各変更ごとにブラウザで動作確認**する。まとめて複数変更を入れてからの確認は禁止

### 禁止事項

- `vite build` の成功だけを根拠にデプロイすること
- ブラウザで動作確認せずに本番デプロイすること
- 動作確認できていない変更を「たぶん大丈夫」でデプロイすること
- 本番が壊れた状態で追加のデプロイを重ねること（まず動く状態にリバートしてからやり直す）

## エージェントへの変更委任

- エージェントが変更したファイルは、マージ前に必ず自分で読んで内容を確認する
- エージェントの「完了しました」を鵜呑みにしない。ビルドと動作確認は自分で行う

## Agent Teams の運用（厳守）

### チーム構成

実装タスクでは以下のチームを常に組織し、自分（Claude）は**指揮・整理・報告**に徹する。コードの直接編集は行わず、必ず Eng チームに依頼する。

| 役割 | 人数 | 責務 |
|------|------|------|
| **Eng A/B/C** | 2-3名 | コード実装。並行作業可能なタスクを分担 |
| **QA Reviewer** | 1名 | Eng の成果物をコードレベルで検証。PASS/FAIL を報告 |
| **UI/UX Reviewer** | 必要時 | デザイン・操作性の観点でレビュー |
| **Music/AI Reviewer** | 必要時 | 音楽的正確性・AI生成品質をレビュー |

### 自分（Claude）の役割

- **やること**: タスク整理、チーム編成、Eng への具体的な作業指示、結果の統合・報告
- **やらないこと**: 自分でコードを直接編集する（Eng に依頼する）
- Eng の作業完了後は必ず QA Reviewer に検証を依頼する
- QA PASS 後にユーザーに報告する

### Eng への作業指示テンプレート

- 問題の背景と原因
- 修正すべきファイルとライン番号
- 期待する修正内容（コード例があれば含める）
- 修正後に `svelte-check --threshold error` を実行する指示

### Eng の修正品質ルール（厳守）

- **修正対象ファイルだけでなく、関連するファイルを必ず読み込んでから修正する。** 1ファイルだけ見て修正すると整合性が崩れる
- 修正前に「この変更が影響する箇所」を列挙し、全て確認してから着手する
- 同じフィールド名・関数名が他のファイルで使われていないか grep で確認する
- 型定義の変更は、その型を使う全ファイルへの影響を確認する
- 「コンパイルが通った」だけでは不十分。ランタイムの挙動を論理的にトレースして検証する

## フォールバック・後方互換性の禁止（厳守）

- フォールバック処理を実装しない。正しいパスのみ実装する
- 後方互換性のための分岐を入れない。古いフォーマットは切り捨てる
- 「念のため」のコードは不要。動かないなら壊れたことを明示する

## バックエンド再起動

- バックエンドのコード変更後は、Claude が自分で再起動する
- ユーザーに再起動を依頼しない

## フロントエンド変更の原則

- CSS-onlyの変更とJS変更は分けて適用する
- JS変更は1ファイルずつ適用→ビルド→動作確認のサイクルで進める
- Svelte 5のrunes (`$state`, `$derived`等) と既存のprop名が衝突しないか注意する
