# kuma-lab プロジェクトルール

## デプロイ前の動作確認（厳守）

**ビルド成功はデプロイ許可ではない。** `vite build` や `dotnet build` の成功は、ランタイムエラーがないことを保証しない。

### デプロイ前に必ず行うこと

1. **ローカルでdev serverを起動**し、ブラウザで対象ページを開いて**画面が表示されること**を目視確認する
2. dev serverのHMRキャッシュが壊れている場合は `.svelte-kit` を削除して再起動し、**新しいブラウザタブ**で確認する
3. ブラウザで確認できない場合（真っ暗になる等）は、**絶対にデプロイしない**。原因を特定してから進める
4. 変更を1つずつ適用し、**各変更ごとにブラウザで動作確認**する。まとめて複数変更を入れてからの確認は禁止

### 禁止事項

- `vite build` の成功だけを根拠にデプロイすること
- ブラウザで動作確認せずに本番デプロイすること
- 動作確認できていない変更を「たぶん大丈夫」でデプロイすること
- 本番が壊れた状態で追加のデプロイを重ねること（まず動く状態にリバートしてからやり直す）

## エージェントへの変更委任

- エージェントが変更したファイルは、マージ前に必ず自分で読んで内容を確認する
- エージェントの「完了しました」を鵜呑みにしない。ビルドと動作確認は自分で行う

## フロントエンド変更の原則

- CSS-onlyの変更とJS変更は分けて適用する
- JS変更は1ファイルずつ適用→ビルド→動作確認のサイクルで進める
- Svelte 5のrunes (`$state`, `$derived`等) と既存のprop名が衝突しないか注意する
