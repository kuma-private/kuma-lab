# UX Audit: コード進行エディタ イベントハンドリング

## 1. イベントハンドリング評価

### ScoreEditor（左のエディタ）

| イベント | 操作 | 評価 | 理由 |
|---------|------|------|------|
| クリック → 編集モード切替 | 表示モードからの遷移 | **x** | 表示モードと編集モードの2状態が存在すること自体がユーザーにとって不透明。クリックしても見た目の変化が小さいと、モード切替が起きたことに気づかない |
| 入力 → onchange | テキスト編集 | **o** | 標準的な入力ハンドリング |
| フォーカスアウト → 表示モードに戻る | 編集終了 | **△** | 意図せずフォーカスが外れるケースが多い（五度圏クリック、ツールバー操作等）。フォーカスアウトのたびにモードが切り替わると操作の連続性が断たれる |
| 右クリック → 選択範囲を再生 | コンテキストメニュー | **o** | パワーユーザー向け機能として適切 |
| ダブルクリック → コード選択+試聴 | 試聴 | **o** | 直感的。ダブルクリックで「選択」はテキストエディタの標準動作に一致 |
| ドロップ → コード挿入 | D&D受け入れ | **△** | ドロップ位置の視覚フィードバック（カーソルインジケータ）がないと、どこに挿入されるか不明確 |

### 五度圏（右サイドバー）

| イベント | 操作 | 評価 | 理由 |
|---------|------|------|------|
| 左クリック → 最後のカーソル位置に挿入 | コード挿入 | **x** | 最大の問題点。「最後のカーソル位置」が視覚的に示されないため、ユーザーはどこに挿入されるか予測できない |
| 右クリック → 試聴 | 音の確認 | **△** | 右クリック=試聴は発見しづらい。OS のコンテキストメニューと競合するリスクもある |
| ドラッグ → エディタにドロップ | D&D | **x** | SVG要素のドラッグは技術的に不安定であり、ユーザーが期待するD&D体験を提供できていない |
| クオリティボタン → トグル | コード種別切替 | **o** | トグル状態が明示されていれば問題なし |
| onボタン → 分数コードモード | モード切替 | **△** | 「on」だけではラベルが不明瞭。分数コードモードが何を意味するか初見で伝わりにくい |

## 2. 主要な問題と代替案

### 問題1: 表示モード/編集モードの二重構造（深刻度: 高）

**現状の問題:**
- ユーザーは「クリックしたら編集できる」ことを知らない
- 表示モードと編集モードの見た目の差が小さいと気づかれない
- フォーカスアウトで意図せずモードが戻る

**代替案: 常時インライン編集 + リッチプレビュー分離**

エディタを常にtextareaとして編集可能にし、別途プレビューエリアでリッチ表示（色付きコードチップ）をリアルタイムに反映する。

```
+-----------------------------+
| [プレビュー] リアルタイム表示  |  <- 色付きコードチップ（読み取り専用）
| | Am7 | Dm7 | G7 | Cmaj7 | |
+-----------------------------+
| [エディタ] 常時編集可能       |  <- textarea、常にフォーカス可能
| Am7 | Dm7 | G7 | Cmaj7     |
+-----------------------------+
```

あるいは、DESIGN.md の設計にあるように「テキスト入力 → リアルタイムでコードチップに変換してプレビュー」方式を徹底する。入力エリアは常にテキスト入力を受け付け、その上にプレビューが表示される。モード切替は不要。

### 問題2: 五度圏からの挿入位置が不明確（深刻度: 高）

**現状の問題:**
- 「最後のカーソル位置に挿入」は、カーソル位置が視覚化されていないと破綻する
- ユーザーの不満「どこに入るかわからない」の直接原因

**代替案A: カーソルインジケータの常時表示**

エディタ内のカーソル位置（挿入ポイント）を、編集モードでなくても視覚的に表示する。点滅するキャレットまたはハイライトされた挿入マーカーを設置。

```
Am7 | Dm7 | G7 |▮| Cmaj7
                 ^-- ここに次のコードが入る
```

**代替案B: 末尾追加を基本動作にする**

五度圏からのクリック挿入は常に「末尾に追加」とし、並べ替えや途中挿入はエディタ内のテキスト編集で行う。これによりユーザーの予測可能性が格段に上がる。

**代替案C: 挿入前プレビュー（推奨）**

五度圏でコードをクリックすると、エディタ上の挿入予定位置にゴースト表示（半透明のプレビュー）が現れ、確定操作（Enter or 再クリック）で挿入される。

```
Am7 | Dm7 | [G7 ゴースト] | Cmaj7
```

### 問題3: 右クリック試聴の発見性の低さ（深刻度: 中）

**代替案: ホバー時の試聴ボタン表示**

五度圏の各コードにホバーすると小さな再生アイコン（スピーカーアイコン）が表示され、それをクリックすると試聴できる。左クリックは挿入、試聴ボタンは試聴、と機能が明確に分離される。

```
   [Am]
   hover -> [Am ▶]  <- ▶をクリックで試聴、Am本体クリックで挿入
```

### 問題4: SVGドラッグの不安定さ（深刻度: 中）

**代替案: ドラッグを廃止し、クリック挿入に一本化**

SVGのD&Dは技術的制約が大きい。ドラッグ機能を削除し、クリックによる挿入（問題2の改善と組み合わせ）に集中する。ユーザーは「五度圏でコードを選ぶ → エディタに入る」というシンプルなフローだけを覚えればよい。

## 3. 総合的なUI改善提案

### 提案1: 挿入フローの単純化

現在の挿入フローは3種類ある（クリック挿入、ドラッグ&ドロップ、テキスト入力）。これを以下のように整理する:

- **主要フロー**: テキスト入力（常時利用可能なtextarea）
- **補助フロー**: 五度圏クリック → 末尾追加（or カーソル位置にゴースト挿入）
- **廃止**: SVGドラッグ&ドロップ

### 提案2: 挿入位置の視覚化

エディタ内に「次のコードがここに入る」マーカーを常時表示する:

- textarea内のカーソル位置と連動するハイライトをプレビュー上に表示
- 五度圏からの挿入時、プレビュー上で挿入位置が一瞬ハイライト（アニメーション）
- 挿入完了時に短いフラッシュアニメーションで「ここに入った」を伝える

### 提案3: モードレスな編集体験

表示/編集モードの切り替えを廃止し、以下のレイアウトにする:

```
+--------------------------------------------------+
|  [ツールバー: | 改行 行削除 ...]                   |
+--------------------------------------------------+
|  プレビュー (リアルタイム、色付きコードチップ)       |
|  | Am7 | Dm7 | G7 | Cmaj7 |                      |
|  | Fmaj7 | Bm7b5 | E7 | Am7 |                   |
+--------------------------------------------------+
|  入力エリア (常時編集可能な textarea)               |
|  Am7 | Dm7 | G7 | Cmaj7                           |
|  Fmaj7 | Bm7b5 | E7 | Am7                        |
+--------------------------------------------------+
```

- プレビューは読み取り専用、入力エリアは常時書き込み可能
- フォーカスアウトでモードが変わることはない
- DESIGN.md に記載の「テキスト入力 → リアルタイムでコードチップに変換してプレビュー」と一致する設計

### 提案4: 五度圏のインタラクション再設計

| 操作 | 現在 | 改善後 |
|------|------|--------|
| 左クリック | カーソル位置に挿入 | 末尾に追加（or ゴーストプレビュー付き挿入） |
| 右クリック | 試聴 | 廃止（ホバー試聴ボタンに移行） |
| ドラッグ | エディタにドロップ | 廃止 |
| ホバー | なし | 試聴ボタン表示 + コード音名のツールチップ |
| 長押し(モバイル) | なし | 試聴（モバイル対応） |

### 提案5: フィードバックの強化

すべての挿入操作に対して、以下のフィードバックを追加:

1. **視覚**: 挿入されたコードチップが一瞬ハイライト（`scale(1.05)` + `box-shadow` アニメーション、300ms）
2. **聴覚**: 挿入時に短い確認音（コードの音を100ms再生）、これは設定でオフにできるようにする
3. **位置**: プレビュー上で挿入位置までスムースにスクロール

### 提案6: ツールバーの改善

- 「行削除」は「最後の行を削除」ではなく、カーソルがある行を削除する動作に変更
- Undo/Redo ボタンを追加（Ctrl+Z / Ctrl+Shift+Z のキーボードショートカットも）
- ツールバーにコードの挿入位置を示すテキスト（例: "行2, 位置5"）を表示

### 提案7: onボタン（分数コード）のラベル改善

- 「on」→「/（分数コード）」に変更
- ツールチップで「分数コード（例: Am7/G）を入力するモードに切り替えます」と説明
- 有効時はボタンの色を変え、五度圏の表示が「ベース音選択モード」に変わることを明示

| 改善項目 | 影響度 | 実装コスト | 優先度 |
|---------|--------|-----------|--------|
| 挿入位置の視覚化（カーソルマーカー） | 高 | 中 | **P0** |
| モードレス編集（表示/編集モード廃止） | 高 | 中 | **P0** |
| 五度圏クリック → 末尾追加にデフォルト変更 | 高 | 低 | **P0** |

| ホバー試聴ボタン追加 | 中 | 中 | **P1** |
| 挿入フィードバックアニメーション | 中 | 低 | **P1** |

## 5. まとめ

ユーザーの不満の根本原因は **「操作の結果が予測できない」** ことに集約される。特に:

1. **挿入位置の不透明さ** -- 五度圏クリック時にどこに入るかわからない
2. **隠れたモード** -- 表示/編集モードの存在がユーザーに伝わっていない
3. **発見しづらい機能** -- 右クリック試聴、ドラッグ&ドロップ

改善の原則は **「1つの操作、1つの結果、1つのフィードバック」**。操作の多様性よりも、各操作の結果の予測可能性を優先すべきである。DESIGN.md に記載されている「テキスト入力 → リアルタイムプレビュー」の設計思想に立ち返り、テキスト入力を主軸に据えたモードレスな編集体験を構築することを推奨する。
